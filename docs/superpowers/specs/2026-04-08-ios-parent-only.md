# iOS Parent-Only Version (#102)

## Overview

Add iOS support for PearGuard as a parent-only device. The iOS parent pairs with Android child devices via P2P, receives usage data, sets policy, and shows notifications. Child enforcement remains Android-only (iOS lacks Accessibility Service / DeviceAdmin APIs).

## Approach

Shared codebase with `Platform.OS` checks in `app/index.tsx`. No separate repo or entry file. The Bare worklet and WebView UI are platform-agnostic and require no changes.

## iOS Native Modules (Swift)

Port from PearCal's existing Swift modules, adapted for PearGuard:

| Module | Purpose | PearCal source |
|--------|---------|---------------|
| `NotificationsModule.swift` | Local notifications: time requests, bypass alerts, PIN overrides, app install/uninstall | NotificationsModule |
| `HapticModule.swift` | UIImpactFeedbackGenerator for button taps | HapticModule |
| `DeepLinkModule.swift` | Handle `pear://pearguard/join/...` invite URLs + notification deep links | DeepLinkModule |
| `ParentBackgroundSync.swift` | BGAppRefreshTask to wake Bare worklet periodically (~15min) for P2P sync | BareBackgroundSync |
| `ShareModule.swift` | Share sheet for invite links | ShareModule |

Each module: `.swift` implementation + `.m` bridging header.

No UsageStatsModule equivalent - parent receives all child data over P2P.

## app/index.tsx Changes

Minimal platform branching:

1. Add `const isAndroid = Platform.OS === 'android'` helper
2. Wrap child-only `DeviceEventEmitter` listeners (`onUsageFlush`, `onBlockOccurred`) in `if (isAndroid)`
3. Platform-branch parent startup: Android calls `UsageStatsModule.startParentService()`, iOS calls `ParentBackgroundSync.register()`
4. Platform-branch notification calls: Android uses `UsageStatsModule.showXNotification()`, iOS uses `NotificationsModule`
5. Platform-branch haptics: Android uses `UsageStatsModule.hapticTap()`, iOS uses `HapticModule.hapticTap()`
6. Child-only paths (`checkChildPermissions`, `clearChildState`, bypass detection) already gated by `_mode === 'child'` - no changes needed
7. App sync (`getInstalledPackages`) only fires from child-initiated events - no changes needed

Most Android-native calls are already optional-chained (`?.`), so the actual diff is small.

## Bare Worklet & UI

No changes. Both are platform-agnostic:

- `src/bare.js` / `src/bare-dispatch.js` - Hyperswarm, Hyperbee, signing, message dispatch work identically regardless of parent OS
- `src/ui/*` - React in WebView. Parent screens (Dashboard, Settings, UsageReports, AppsTab, RulesTab, ActivityTab) render the same everywhere

## Xcode Project & Configuration

1. Generate `ios/` directory via `npx expo prebuild --platform ios`
2. Add `ios` block to `app.json`:
   - `bundleIdentifier`: `com.pearguard`
   - `buildNumber`: `1`
   - `infoPlist`: camera/photo permissions, background modes (`fetch`, `processing`), URL scheme (`pear`)
3. Podfile auto-generated by Expo; add pods for `react-native-bare-kit`, `react-native-webview`, etc.
4. Swift modules in `ios/PearGuard/`, registered via bridging pattern
5. `Info.plist`: `BGTaskSchedulerPermittedIdentifiers` for background sync task
6. Automatic Xcode signing initially; production signing deferred to release pre-work

## Scope

**In scope:**
- iOS parent pairs with Android child devices via P2P (invite links, QR codes)
- Receives and displays usage reports, app lists, alerts
- Sets and syncs policy/rules to child devices
- Local notifications for time requests, bypass alerts, PIN overrides, app events
- Background fetch for periodic P2P sync (~15min intervals, iOS-controlled)
- Co-parent pairing (iOS parent + Android parent managing same child)

**Out of scope / known limitations:**
- No iOS child mode - enforcement APIs don't exist on iOS
- No real-time connectivity when backgrounded - only periodic background fetch windows
- Parent must open the app for guaranteed immediate sync
- No equivalent to Android's ParentConnectionService persistent foreground service
- Existing "last seen" timestamps on Dashboard already communicate sync staleness - no special iOS UI needed
