### P2P delivery reliability (first time request silently lost)
Root cause: `sendToParent` in `bare.js` wrote to a potentially stale TCP connection (dropped while app was backgrounded, `close` event not yet fired). The write appeared to succeed but data was lost. Message was NOT queued, so reconnect didn't re-deliver it.

**Fix**:
- `src/bare.js` `sendToParent`: Now always queues the message in Hyperbee first, then attempts immediate delivery. If connection is live, message is sent immediately AND stays in queue as backup. On next reconnect, `handleHello` → `flushPendingMessages` re-delivers it.
- `src/bare-dispatch.js` `handleIncomingTimeRequest`: Dedup check — if `request:{requestId}` already exists on parent, silently returns. Prevents duplicate notifications when re-delivered message arrives.
- New test: "duplicate: does not re-emit event or overwrite existing entry"

**Observed behavior after fix**: First request is delivered after parent foregrounds the app (triggers reconnect). Queue flush happens during hello exchange. Working correctly.
